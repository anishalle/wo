# `wo` CLI Tool - Technical Analysis and Implementation Report

`wo` is a fast workspace manager for repo-style directories written in Go. It enables users to jump into workspaces, browse indexed projects via an interactive TUI or FZF, and execute workspace-specific "enter hooks".

## 1. Overall Architecture

The application is structured into clearly separated packages:
- **`cmd/`**: Entry point for the CLI.
- **`internal/cli/`**: CLI command definitions (using Cobra) and orchestration.
- **`internal/config/`**: Configuration parsing (`config.toml`), defaults, and normalization.
- **`internal/db/`**: Persistence layer using SQLite (`modernc.org/sqlite`).
- **`internal/model/`**: Shared core domain models (e.g., `Workspace`, `ResolveResponse`).
- **`internal/resolve/`**: Search and matching engine (exact, prefix, fuzzy matching, and correction).
- **`internal/scan/`**: Filesystem traversal to discover git repositories and `.wo` directories.
- **`internal/hooks/`**: Security and execution of workspace entry hooks.
- **`internal/tui/`**: Interactive pickers utilizing Bubble Tea and Lip Gloss, with an alternative path for `fzf`.
- **`internal/shell/`**: Shell integration scripts (Bash, Zsh, Fish) that enable the `wo` command to change directories and evaluate scripts dynamically.

## 2. Design Patterns Used

1. **Dependency Injection & App Context (`cli.App`)**
   - The root command setup initializes an `App` struct holding the initialized `Config` and `db.Store`. This struct is injected into context (`cmd.SetContext`), which commands extract. This makes unit testing and resource lifecycle management straightforward.
2. **Repository Pattern (`db.Store`)**
   - Database operations are abstracted away in methods like `UpsertWorkspace`, `ListWorkspaces`, `SetTrust`, and `UsageMap`. The rest of the application interacts with domain objects (`model.Workspace`) rather than raw SQL.
3. **Model-View-Update (MVU) Pattern (`tui` package)**
   - The TUI uses the Bubble Tea framework. State is managed via `modelPicker`, commands via `tea.Cmd`, and the view is generated by combining standard list components with custom styling from Lip Gloss.
4. **Strategy / Pipeline Pattern (`resolve.Service`)**
   - The resolution engine evaluates a user query sequentially against multiple strategies: Alias matching -> Exact match -> Prefix match -> Fuzzy matching. The system cascades down until a threshold or match is met.
5. **Interceptor / Hook Pattern (`hooks.Service`)**
   - Workspace transitions intercept standard flow to check for "enter" hooks defined in `.wo` files. They utilize a Trust decision layer (prompting the user and storing decisions in SQLite) to prevent unauthorized execution.

## 3. Implementation Rating & Feedback

### Strengths & Positive Highlights (Rating: 8.5/10)

- **Performance-Oriented Persistence**: Uses `PRAGMA journal_mode=WAL;` for SQLite. This significantly improves read performance and concurrency, which is excellent for a fast CLI tool executing on every shell prompt.
- **Excellent Domain Separation**: The separation into distinct `internal` packages makes the codebase very clean. It prevents circular dependencies and separates the database mapping from the domain logic.
- **Smart Resolution & Fuzzy Matching**: The `resolve.similarity` implementation is robust. It combines Subsequence matching and Levenshtein distance, scaling the score smoothly. The built-in correction mechanism (`shouldSuggestCorrection`) is a great UX touch.
- **Security-First Execution**: The implementation of `wo trust` and the tracking of workspace fingerprints before executing arbitrary `.wo` shell commands is a well-thought-out security model.
- **Elegant Shell Integration**: Embedding the shell integrations natively inside Go (via `shell` package constants) avoids the need to distribute separate `.sh` files. Using a dynamic eval pattern (`eval "$(wo __shell-apply...)"`) correctly allows the CLI tool to modify the parent shell environment securely.

### Weaknesses & Areas for Improvement

- **SQL String Concatenation (`DeleteMissingWorkspaces`)**:
  In `db/store.go`, the query for deletion dynamically generates placeholders (`strings.Repeat("?,", len(keepPaths))`). While safe due to parameter binding, SQLite has a hard limit on variables per statement (SQLITE_MAX_VARIABLE_NUMBER, default 999 or 32766). For massive filesystem scans, this could crash. Batching the deletes would be safer.
- **Error Handling Granularity**:
  Most errors cascade out directly (`return out, err`). Creating centralized custom errors (e.g., using `errors.Is(err, ErrWorkspaceNotFound)`) would allow the CLI layer to print more user-friendly messages rather than bare underlying system errors.
- **Shell Script Maintenance**:
  The bash/zsh/fish scripts are hardcoded as raw strings in `shell/init.go`. While convenient for distribution, this breaks IDE syntax highlighting and linting for the bash code. Moving them to actual `.sh` and `.fish` files and utilizing `go:embed` would be cleaner.
- **Test Coverage Context**:
  Most core logic like fuzzy resolution and database schema has test coverage, but it's largely focused on positive execution flow. Adding more robust failure scenario testing for the CLI orchestration would elevate the quality.

## Conclusion

The `wo` CLI tool is extremely well-designed. The code is modular, performance-centric, and incorporates advanced CLI UX features like fuzzy finding, interactive pickers, and security context tracking. The codebase successfully employs standard Go best practices. With a few minor robustness improvements around database scaling limits and string-based scripts, it is a production-ready utility.
